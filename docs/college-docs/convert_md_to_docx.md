# From Markdown To Professional DOCX

Use this skill when converting a finalized Markdown report into a polished Word document. The goal is not only to preserve Markdown structure, but to create a professional client-ready `.docx` with clean typography, report-style headings, readable tables, page numbers, and validation.

The reusable implementation lives in:

```text
tools/markdown_to_professional_docx.py
```

Do not copy the Python implementation into this Markdown file. Keep the executable logic in `tools/markdown_to_professional_docx.py` and keep this skill file as the operational guide.

## Core Approach

Use a two-step pipeline implemented by the tool:

1. Convert Markdown to DOCX with Pandoc.
2. Reopen the generated DOCX with `python-docx` and apply professional styling.

Pandoc handles document structure:

- Markdown headings become Word headings.
- Markdown tables become Word tables.
- Markdown bullet and numbered lists become Word lists.
- Markdown bold/italic formatting is preserved.

The `python-docx` post-processing step provides the polished result:

- Removes duplicate title generated by Pandoc metadata plus Markdown H1.
- Sets page margins.
- Applies consistent fonts and colors.
- Styles headings.
- Cleans the front-matter block.
- Styles tables and table headers.
- Repeats table headers across pages.
- Uses smaller font for very large detailed-result tables.
- Adds page-number footer.
- Validates the generated file.

By default, the tool does **not** request Word to auto-update fields on open, because that setting can trigger the Word prompt:

```text
This document contains fields that may refer to other files.
Do you want to update the fields in this document?
```

There are two independent triggers for this prompt, and the tool addresses both by default:

1. The document-level `w:updateFields` setting in `word/settings.xml`. The tool removes that element so Word does not auto-update fields on open.
2. The per-field `w:dirty="true"` attribute that Pandoc adds to its generated TOC `fldChar`. Even with the document-level setting removed, this per-field flag alone is enough to make Word show the prompt every time the file opens. The tool strips `w:dirty` from every `fldChar` in the body, headers, and footers as part of post-processing.

If automatic TOC refresh on open is more important than avoiding that prompt, opt in with `--update-fields-on-open`. In that mode the tool sets `w:updateFields=true` and leaves the per-field dirty flags in place, so Word refreshes the TOC on open (and shows the prompt). Without that flag, the user can still refresh the TOC manually inside Word with Ctrl+A then F9.

## Prerequisites

Prefer this stack:

- `pandoc` available on PATH.
- Python available on PATH.
- `python-docx` installed.

Check availability:

```powershell
Get-Command pandoc -ErrorAction SilentlyContinue | Select-Object Source,Version
Get-Command python -ErrorAction SilentlyContinue | Select-Object Source,Version
python -c "import docx; print(docx.__version__)"
```

If `python-docx` is missing and installing packages is allowed, install it in the active project environment. If installing is not allowed, use Pandoc-only conversion and clearly state that the polished post-processing could not be applied.

## Primary Command

Use the repository tool for the full conversion:

```powershell
python tools\markdown_to_professional_docx.py path\to\report.md
```

Specify the output path and title when needed:

```powershell
python tools\markdown_to_professional_docx.py path\to\report.md `
  --output path\to\report.docx `
  --title "Report Title"
```

Useful options:

```text
--output PATH             Output DOCX path. Defaults to input path with .docx extension.
--title TEXT              DOCX metadata/title. Defaults to first Markdown H1 or file name.
--from-format FORMAT      Pandoc input format. Default: gfm.
--no-toc                  Do not generate a table of contents.
--toc-depth N             Table-of-contents depth. Default: 2.
--reference-doc PATH      Optional Pandoc reference DOCX template.
--post-process-only       Treat input as an existing DOCX and only apply professional styling.
--skip-validation         Skip DOCX structural validation after generation.
--update-fields-on-open   Ask Word to auto-update TOC/other fields on open.
```

Use `--post-process-only` when Pandoc conversion has already been completed:

```powershell
python tools\markdown_to_professional_docx.py path\to\report.docx --post-process-only
```

## Pandoc Conversion Details

The tool runs Pandoc internally using this pattern:

```powershell
pandoc path\to\report.md `
  --from=gfm `
  --to=docx `
  --resource-path="<markdown-folder>[;<invocation-folder>]" `
  --toc `
  --toc-depth=2 `
  --standalone `
  --metadata title="Report Title" `
  --output=path\to\report.docx
```

Use `--from=gfm` for GitHub-flavored Markdown tables and common Markdown syntax.

The repository tool also runs Pandoc from the Markdown file's own directory, so relative image paths are resolved from the document location rather than from the shell's current folder.

Use a TOC for substantial reports. For short documents, pass `--no-toc`.

Because the tool avoids `updateFields` by default, a Word TOC may require a manual update inside Word after opening if you want the final page numbers refreshed without using `--update-fields-on-open`.

Use `--reference-doc` only when the user provides or requests a corporate template. The tool still applies the professional post-processing after Pandoc finishes.

## Front-Matter Cleanup

If the first few paragraphs include report metadata such as:

```text

```

The tool rebuilds that front-matter paragraph with explicit line breaks:

```text
Prepared for: ...
```

The tool centers the front-matter paragraph and adds extra spacing after it.

## Table Styling Rules

The tool applies these settings to every table:

```text
Style: Table Grid
Alignment: centered
Width: 100%
Autofit: true
Cell vertical alignment: top
Cell padding: small and consistent
```

The tool applies these settings to the first row of every table:

```text
Header background: RGB hex #1F4E79
Header font color: white
Header font weight: bold
Header repeats across pages: true
```

Table font sizes:

```text
Normal tables: 9 pt
Large detailed-result table: 7.5 pt
```

The tool detects a detailed test-results table by checking for this header:

```text
No. | Type | Title | Outcome | Comment
```

For that table only, it uses 7.5 pt so long titles and comments fit better in Word.

## Footer

The tool adds a centered footer:

```text
Page {PAGE}
```

It uses a real Word PAGE field, not plain text. Page numbering works without enabling document-wide auto-update on open, because the PAGE field is rendered by Word at print/layout time and is not marked dirty by the tool.

The tool does **not** enable document-wide field updates on open by default, and it also strips the per-field `w:dirty` attribute that Pandoc adds to the TOC. Both steps together prevent the "This document contains fields that may refer to other files" prompt for client-facing reports. Enable auto-update behavior only with `--update-fields-on-open`, which both sets `w:updateFields=true` and keeps the dirty flags so Word refreshes the TOC on open.

## Validation

The tool validates the generated DOCX by default:

- Opens the file with `python-docx`.
- Checks the DOCX ZIP package.
- Confirms `word/document.xml` exists.
- Prints paragraph and table counts.

Manual validation commands, if needed:

```powershell
python -c "from docx import Document; p=r'path\to\report.docx'; d=Document(p); print('paragraphs', len(d.paragraphs), 'tables', len(d.tables)); print([(i, para.text, para.style.name) for i, para in enumerate(d.paragraphs[:8]) if para.text.strip()])"
```

```powershell
python -c "import zipfile; p=r'path\to\report.docx'; z=zipfile.ZipFile(p); bad=z.testzip(); print('zip_ok', bad is None); print('document_xml', 'word/document.xml' in z.namelist())"
```

If the document has a large appendix table, verify the table count and inspect the first paragraphs to make sure:

- There is not a duplicate title.
- The front-matter block has clean line breaks.
- Headings are still mapped to heading styles.
- Tables are present.

## Practical Notes

- If the user says they like the previous professional DOCX style, use this tool.
- If the report contains very wide tables, keep them in Word as tables but reduce font size for those tables. The tool already does this for detected detailed-result tables.
- If a table remains too wide, ask whether to split it, move it to an appendix, or use landscape orientation for that section. Do not make that structural change without user approval unless explicitly requested.
- If the document needs a corporate template, use `--reference-doc`.
- If the user moved the DOCX outside the repo, generate the new DOCX in the requested destination and verify it there.
- Prefer the default no-auto-update behavior for client-facing documents to avoid the Word field-update prompt on open.
- If the user explicitly wants the TOC to refresh automatically on open and accepts the Word prompt, use `--update-fields-on-open`.
- Do not claim visual perfection without opening or validating the DOCX structure. At minimum, validate with the tool's default validation output or the manual validation commands above.
