# DDR-017: стандартизовані соціальні прев’ю сторінок

## Статус
Прийнято

## Дата
2026-06-30

## Контекст
Коли користувач надсилає посилання на сторінку сайту в месенджер або соціальну мережу, сервіс зазвичай показує прев’ю: картинку, заголовок і короткий опис. Для сайту коледжу потрібно, щоб такі прев’ю виглядали стабільно, офіційно й не вимагали окремої редакційної роботи для кожної сторінки.

Раніше в ТЗ це було позначено як точку для майбутнього редакційного затвердження. 2026-06-30 підтверджено, що базова модель має бути стандартизованою для всіх сторінок.

## Рішення
- Усі публічні сторінки сайту мають містити соціальні metadata: `meta name="description"`, Open Graph-теги й Twitter/X card-теги.
- У межах кожної мовної версії всі сторінки використовують однакові соціальні `title`, `description` і `image`.
- Для української версії затверджений заголовок прев’ю: «Економіко-правовий фаховий коледж ККІБП».
- Для української версії затверджений опис прев’ю: «Офіційний сайт коледжу: вступ, навчання, студентське життя, новини та корисні сервіси.»
- Для англійської версії затверджений заголовок прев’ю: «Professional College of Economics and Law, KCIBL».
- Для англійської версії затверджений опис прев’ю: «Official college website: admissions, studies, student life, news and useful services.»
- Зображення прев’ю для всіх сторінок - логотип коледжу `assets/logo_small.gif` або інший окремо затверджений share-image asset.
- `og:url` має залишатися сторінковим і відповідати канонічній адресі конкретної сторінки.
- Браузерний `<title>` може залишатися сторінковим, але `og:title` і `twitter:title` мають використовувати затверджений загальний заголовок.

## Альтернативи
- **Унікальне прев’ю для кожної сторінки**: відхилено як базова модель, бо це збільшує редакційне навантаження і створює ризик нерівномірної якості.
- **Використовувати фото новини як share-image для сторінки новини**: не прийнято для базової моделі. Сайт має показувати стабільний офіційний логотип у прев’ю.
- **Не налаштовувати соціальні metadata**: відхилено, бо посилання в месенджерах виглядатимуть непередбачувано.

## Наслідки
- У майбутній CMS / production-реалізації соціальні metadata мають формуватися централізовано на рівні теми, шаблону або SEO-модуля.
- Редактор не повинен вручну заповнювати social title, social description або share-image для кожної сторінки як обов’язковий щоденний сценарій.
- Згенеровані сторінки окремих новин у статичній демоверсії потрібно перед фінальною передачею синхронізувати з цією моделлю, якщо вони ще містять article-specific social metadata з імпорту.
- Якщо в майбутньому потрібні промо-винятки для окремих сторінок, їх потрібно погоджувати окремим рішенням.
- Під час review потрібно перевіряти прев’ю для головної сторінки, сторінки новин, сторінки окремої новини й англомовної сторінки: заголовок, опис і зображення мають бути однаковими в межах мовної версії, а `og:url` має бути сторінковим.
