# DDR-004: глобальний header і footer у статичній демоверсії

## Статус
Прийнято

## Дата
2026-06-29

## Контекст
Демоверсія сайту публікується через GitHub Pages, тому фінальний результат має залишатися набором готових статичних HTML/CSS/JS-файлів. Водночас header і footer є глобальними елементами інтерфейсу: якщо їх редагувати вручну в кожному HTML-файлі, легко отримати розсинхрон між сторінками.

Після додавання CTA “Вступ” у header і швидких переходів у footer було виявлено дві розбіжності: `news.html` мав коротший nav-блок header, а швидкі переходи footer були тільки на головній українській сторінці.

## Рішення
- Header і footer мають бути однаковими для всіх сторінок у межах кожної мовної версії.
- Допустимі відмінності: активний стан поточної сторінки, мовні тексти та відносні шляхи до ресурсів.
- У header після пункту “Новини” / “News” показується виділена CTA-кнопка “Вступ” / “Admissions”.
- Кнопки верхніх дій header мають бути візуально уніфікованими. На desktop і tablet кнопки пошуку, перемикання теми та перемикання мови мають мати однаковий квадратний розмір і однаковий радіус заокруглення в межах одного header-рядка.
- У footer під брендовим блоком на всіх сторінках показується група швидких переходів без видимого заголовка.
- Статична демоверсія для GitHub Pages залишається готовими HTML-файлами, але ці файли мають синхронізуватися з одного джерела в локальному скрипті, а не редагуватися вручну як незалежні копії.
- У майбутній WordPress-реалізації header і footer мають бути справжніми глобальними компонентами теми.

## Альтернативи
- **Залишити ручне дублювання без синхронізації**: відхилено, бо кожна зміна header/footer створює ризик, що одна сторінка залишиться зі старою структурою.
- **Підвантажувати `header.html` і `footer.html` через JavaScript у браузері**: відхилено для демоверсії, бо це гірше для SEO, доступності, локального відкриття файлів і стабільності першого рендеру.
- **Додати build-систему з окремим деплоєм**: не потрібно для поточного масштабу демоверсії. Достатньо локального скрипта, який оновлює готові HTML перед публікацією.

## Наслідки
- Для GitHub Pages у репозиторії залишаються готові статичні `.html` файли.
- Header і footer можуть фізично бути присутні в кожному HTML-файлі, але їхня структура має походити з одного локального джерела синхронізації.
- Швидкі переходи footer більше не є особливістю лише головної сторінки.
- Перед передачею у WordPress потрібно вимагати централізований `header` і `footer`, керовані темою або компонентами, а не ручним дублюванням на сторінках.
- Під час додавання нових header-actions не можна вводити окремі розміри для мовного перемикача або інших службових кнопок без нового дизайн-рішення.

## Перевірка
- Після синхронізації потрібно перевіряти всі основні українські та англійські сторінки.
- Обов’язково перевіряти mobile, tablet, laptop і wide viewport на відсутність горизонтального переповнення.
- Окремо перевіряти, що `news.html` має ті самі nav-tools у header, що й інші українські сторінки.
- Окремо перевіряти фактичні розміри header-actions: на desktop/tablet кнопки пошуку, перемикання теми і перемикання мови мають бути однаковими за шириною й висотою.
